---
title: Static Extraction
description: Move supported styles into CSS during production builds.
---

A production app should not do unnecessary style work in the browser.

If a style decision is already known at build time, it can be moved into CSS
assets.

The important part is that component authors should not have to switch APIs just
to get better production output.

Static extraction is a build choice.

The source code can keep reading like React and TypeScript.

You still write:

```txt
style -> slot -> scope -> resolve
```

The compiler extracts supported static chains into CSS assets and leaves
props, conditions, and app-specific composition for runtime resolution.

Production builds get efficient CSS.

The component code still works with TypeScript, debugging, and refactoring in
the same way.

That distinction matters because not every style decision is knowable at build
time.

Static styles are the easy case:

```ts
const button = style({
  border: 0,
  borderRadius: 8,
  padding: '8px 12px',
}).hover({
  opacity: 0.86,
});
```

The compiler can move those rules into CSS because the declarations are known.

Dynamic composition stays in React:

```tsx
export function Button(props: ButtonProps) {
  const css = combineStyle(
    buttonStyles,
    props.danger && dangerButton(buttonStyles.root),
    props.theme?.(buttonStyles.root),
  );

  return <button css={[css.root, props.css]}>{props.children}</button>;
}
```

The build can extract the static parts of `buttonStyles` and `dangerButton`.

The runtime still resolves the conditional parts, incoming themes, and
`props.css`.

In production, both paths meet as class names and token values.

In development, both paths keep source metadata so DevTools can point back to
the authored style, scope, token, or JSX element that produced the rule.
