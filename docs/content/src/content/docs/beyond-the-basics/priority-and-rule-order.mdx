---
title: Priority And Rule Order
description: Control generated rule order when normal composition is not enough.
---

Two generated rules can both be correct.

The problem is deciding which one should win.

A base style, hover state, component theme, media rule, and token override can
all meet on the same property.

Without predictable rule order, developers eventually have to debug generated
CSS instead of the source styles they wrote.

Generated rules need a stable order:

The order considers:

- explicit value weight;
- selector bucket;
- scope bucket;
- media bucket;
- property bucket.

Most users will not tune priority often.

Priority tuning is mostly for design-system authors.

It matters when base styles, component themes, state scopes, media rules, and
token overrides all meet.

For the deeper bucket model, see [Priority Model](../design/priority-model).

Ordering is one generated-CSS concern. The other is deciding which style work
can move out of the browser before the app runs.

Most component code controls priority by composition order:

```tsx
const css = combineStyle(
  buttonStyles,
  props.danger && dangerButton(buttonStyles.root),
  props.compact && compactButton(buttonStyles.root),
  props.theme?.(buttonStyles.root),
);

return <button css={[css.root, props.css]} />;
```

Here the component resolves its base slots, variants, and incoming component
theme first.

The caller's root `props.css` is last because it is the final root override.

Design-system authors can also control state priority in a custom selector
map:

```ts
import { createStyleFn, type CSSProperties, type } from '@fluentic/style';
import { selector, selectorPriority } from '@fluentic/style/selector';

const selectors = {
  hover: selector(':hover'),
  focusVisible: selector(':focus-visible'),
  pressed: selector('[aria-pressed="true"]'),
  disabled: selector(':disabled'),
};

const prioritySelectors = selectorPriority(selectors, [
  'hover',
  'focusVisible',
  'pressed',
  'disabled',
]);

export const { style: ui } = createStyleFn({
  style: type<CSSProperties>,
  selectors: prioritySelectors,
});
```

The list is ordered from lower to higher priority.

In this example, `disabled` wins over `pressed`, `pressed` wins over
`focusVisible`, and `focusVisible` wins over `hover` when they target the same
property.

During development, `StyleDevUtils` can switch priority output:

```ts
StyleDevUtils.setPriorityMode.toLayer();
StyleDevUtils.setPriorityMode.toSort();
```

Layer mode is easier to inspect when you want to see priority buckets.

Sort mode is closer to compact generated output.
