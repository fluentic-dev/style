---
title: Styling Tradeoffs
description: Common CSS-in-JS and extraction tradeoffs, and how Fluentic Style approaches them.
---

Styling tools sit on top of CSS, so they inherit CSS behavior first. Then they
add their own choices about authoring, ordering, extraction, runtime work, and
debugging. Fluentic Style is built around those tradeoffs instead of pretending
they do not exist.

The goal is not to hide CSS. The goal is to give component authors clearer
override points, predictable delivery, and a runtime path when a value is truly
dynamic.

## CSS selectors have sharp edges

Selectors are powerful, but they can become hard to maintain when a component
system grows.

- Source order still matters when specificity is equal.
- Specificity can make a small override unexpectedly difficult.
- Deep descendant selectors couple consumers to internal markup.
- State selectors can drift away from the component part they are meant to
  affect.
- Shorthands and longhands can override each other in ways that are not obvious
  when rules are split apart.

Fluentic keeps selectors close to the style chain:

```ts
const trigger = style.slot({
  borderColor: tokens.color.border,
})
  .hover({
    borderColor: tokens.color.accent,
  })
  .focusVisible({
    outline: `2px solid ${tokens.color.accent}`,
  });
```

For component internals, prefer slots over deep selectors:

```ts
const menu = {
  root: style.slot({ display: 'grid' }),
  item: style.slot({ minHeight: 36 }),
};

const compactMenu = style.scope([
  menu.item({ minHeight: 30 }),
]);
```

That gives consumers a named target instead of asking them to know the generated
class name or internal DOM structure.

## Runtime CSS-in-JS has real costs

Runtime styling is flexible. It is also easy to pay for that flexibility in
places you did not intend.

- Styles may be calculated during render.
- CSS insertion order can depend on render order.
- Server rendering and hydration need coordination.
- Every dynamic object can become a new cache entry.
- DevTools can show many generated classes instead of the component-level
  reason for a style.

Fluentic keeps runtime support because some values only exist at runtime. The
important part is that runtime is not the only delivery path.

```tsx
const css = combineStyle(
  button,
  isActive && tokens.color.accent('#16a34a'),
);
```

Use runtime for values that are actually dynamic. Let the compiler extract
static chains when the value is known ahead of time.

## Atomic CSS needs ordering rules

Atomic CSS keeps output small and reusable by emitting one class per
declaration. That is useful, but atomic output has to answer hard ordering
questions.

- Two classes may write the same property.
- A shorthand and a longhand may both apply.
- Selectors and media rules need stable priority.
- Composition order in TypeScript needs to become source order in CSS.
- Extracted files should not change behavior because the module graph or chunk
  graph changed.

Fluentic treats order as part of the style data. Style chains, selector
priorities, scopes, and `combineStyle` composition all feed the same resolution
model, whether styles are resolved at runtime or extracted into CSS.

```ts
const button = style.slot({
  backgroundColor: tokens.color.surface,
  color: tokens.color.text,
})
  .hover({
    backgroundColor: tokens.color.surfaceRaised,
  });

const danger = style.scope([
  button({ color: tokens.color.danger }),
]);
```

The generated CSS can be atomic, but the authoring model still carries the
component-level intent.

## Extraction can feel random without a model

Build-time extraction is great when it works. The rough edges usually appear
around analyzability and ordering.

- Some extractors require a narrow subset of JavaScript or TypeScript.
- A style may stop extracting when it moves behind a helper function.
- CSS order can follow file traversal, module evaluation, or chunk generation
  instead of component intent.
- Runtime fallback can become an afterthought.

Fluentic's compiler recognizes Fluentic API calls, then lets the surrounding
code remain normal TypeScript. The convention is the API itself:
`style(...)`, `style.slot(...)`, `style.scope(...)`, tokens, themes, and
composition.

```ts
const size = dense ? 32 : 40;

const control = style.slot({
  minHeight: size,
  paddingInline: size / 4,
});
```

Static styles can be extracted. Runtime values can still flow through the
runtime path. The code does not need to become a separate styling language.

## Sourcemaps are useful, but noisy

Generated CSS is not the source of truth. Sourcemaps help, but atomic CSS makes
debugging noisier because one component can produce many small rules.

Common problems include:

- many generated rules pointing at one source file;
- rules pointing at transformed output instead of authored code;
- browser DevTools showing an atomic class instead of the component part;
- production builds removing the nearby context that made a style readable.

Fluentic keeps authored style chains as the mental model. Development builds can
preserve debug metadata, local class information, and CSS sourcemap sources, so
the generated CSS remains traceable without making generated CSS the API.

## What Fluentic proposes

Fluentic Style does not remove CSS complexity. It gives component systems a
clearer place to put it.

- **Named override points:** slots publish the parts consumers may style.
- **Local selector chains:** hover, focus, media, container, and custom
  selectors stay near the component styles they affect.
- **Composable themes:** scopes target slots and can travel through subtrees.
- **Themeable tokens:** values that need identity become tokens; regular
  TypeScript constants stay regular constants.
- **Predictable atomic output:** static chains can extract to atomic CSS while
  preserving Fluentic's ordering model.
- **Runtime fallback:** truly dynamic values can still resolve at runtime.
- **Traceable debugging:** generated CSS is delivery output, while authored
  TypeScript remains the source of truth.

The result is a smaller promise: write TypeScript as TypeScript, expose the
styling API you mean to support, and let Fluentic handle the hard parts:
resolution, themes, tokens, extraction, and delivery.
