---
title: Keeping Styles As Data
description: Keep style declarations as plain data that TypeScript can understand.
---

Most component styles begin as a plain object.

You can move them between files.

You can spread them.

You can inspect them.

TypeScript understands them.

That ordinary shape is useful.

Nested object DSLs are familiar, but they mix several responsibilities:

- declarations;
- selectors;
- state;
- media;
- relationships.

Fluentic splits those responsibilities into separate APIs:

```txt
declarations -> style object
self state and conditions -> chain
relationships -> slots and scopes
render-time merge -> combineStyle
```

That raises the next question: which styleable parts should become public
contracts, and which should remain local data?
