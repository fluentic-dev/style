---
title: Runtime And Extraction Contract
description: Keep custom builders, dynamic React values, and extracted CSS on the same authoring model.
---

Fluentic has to support two frontend realities at the same time.

During development, styles need to behave like normal React values. Components
receive props, themes, and conditional scopes at render time.

In production, static styles should be able to move into CSS assets.

Those needs should not create two different styling languages.

```txt
same authored style value
  -> runtime path when React decides it dynamically
  -> extraction path when the compiler can prove it is static
```

That contract is especially important once a project adds custom builders.
Custom fields, custom chain methods, slots, and scopes still need to resolve in
development and still need a path to extracted CSS.

## Runtime First

The runtime can handle values the build cannot know:

```txt
props
conditional scopes
app themes
component library inputs
dynamic token overrides
```

```tsx
const css = combineStyle(
  buttonStyles,
  props.compact && compact(buttonStyles.root),
  bindScope(buttonStyles.root, props.theme),
);
```

This is normal React work. The component has the context, so the component
resolves the style inputs.

## Extract When Static

The compiler can extract supported static chains:

```ts
const button = ui({
  row: true,
  gapX: 8,
}).hover({
  opacity: 0.88,
});
```

The build can move the generated atomic rules into CSS while leaving dynamic
cases on the runtime path.

For custom builders, the compiler needs to know the imported style function:

```ts
stylePlugin({
  importSources: [
    {
      source: './ui-style',
      name: 'ui',
      styleFn: ui,
    },
  ],
});
```

## The Promise

The authoring model should not change when a project moves from runtime-only
development to extracted production CSS:

```txt
style object    -> builder data
chain method    -> selector/at-rule data
custom field    -> transform to CSS
slot            -> component part identity
scope           -> grouped slot overrides
combineStyle    -> component-boundary resolution
css prop        -> final element output
```

That is the balance Fluentic is trying to keep: fluent APIs for authors,
component-aware runtime composition for React, and extracted CSS when the build
can safely do the work earlier.
