---
title: Tradeoffs
description: The costs Fluentic accepts to keep React styling component-aware, extractable, and traceable.
---

Fluentic chooses more structure than plain class names.

That structure buys component-aware styling, typed composition, static
extraction, and debugging identity.

It also has costs:

| Tradeoff | Cost |
| --- | --- |
| Slots are contracts | Component authors must avoid exposing implementation details. |
| Scopes are a separate concept | Consumers learn the difference between root styles and component themes. |
| Runtime remains | React-known choices cannot all move to build time. |
| Compiler integration matters | Production extraction requires bundler setup. |
| Custom vocabulary is powerful | Bad custom fields can hide ordinary CSS behind local jargon. |
| Debug metadata has modes | Teams choose whether spread/merge traces point to style sites or value sites. |

The design goal is not to remove every decision.

It is to make the important decisions visible:

```txt
what is local
what is public
what can be themed
what resolves at runtime
what can be extracted
what can be traced
```

That is the architecture Fluentic is built around.
