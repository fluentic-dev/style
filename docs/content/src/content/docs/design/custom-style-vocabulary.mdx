---
title: Custom Style Vocabulary
description: Create project-specific style fields and chain methods without leaving Fluentic's runtime and compiler model.
---

A design system often develops words that are not raw CSS properties.

Maybe authors keep writing layout shorthands:

```ts
{
  row: true,
  gapX: 8,
  center: true,
}
```

Maybe components share semantic states:

```ts
style().pressed({ transform: 'translateY(1px)' });
style().selected({ backgroundColor: '#eef2ff' });
```

Those are useful only if they stay honest: they still need to become ordinary
CSS rules, work at runtime, and keep working when the compiler extracts static
styles.

## Style Fields

Custom style fields belong in a transform:

```ts
type LayoutStyle = CSSProperties & {
  row?: boolean;
  center?: boolean;
  gapX?: CSSProperties['columnGap'];
};

const layoutTransform = styleTransform<LayoutStyle>({
  transform(style) {
    const { row, center, gapX, ...css } = style;

    if (row) {
      css.display = 'flex';
      css.flexDirection = 'row';
    }

    if (center) {
      css.alignItems = 'center';
      css.justifyContent = 'center';
    }

    if (gapX !== undefined) css.columnGap = gapX;

    return css;
  },
});
```

The transform gives the project vocabulary a single path back to CSS.

## Chain Methods

Custom states and responsive helpers belong in selectors:

```ts
const selectors = {
  hover: selector(':hover'),
  pressed: selector('[aria-pressed="true"]'),
  below: selector('@media (max-width: $$)', 'media'),
};
```

Then `createStyleFn(...)` ties the style type, transform, selectors, and
priority model together:

```ts
export const { style: ui } = createStyleFn({
  style: type<LayoutStyle>,
  selectors: selectorPriority(selectors, ['hover', 'pressed']),
  transform: layoutTransform,
});
```

Component authors can now write in project language while still producing
normal CSS:

```ts
const button = ui({
  row: true,
  center: true,
  gapX: 8,
  border: 0,
}).hover({
  opacity: 0.88,
}).pressed({
  transform: 'translateY(1px)',
}).below('700px', {
  width: '100%',
});
```

## Slots And Scopes Use The Same Vocabulary

A custom builder is not only for local element styles. Its slots and scopes use
the same transform and selector vocabulary:

```ts
const buttonStyles = {
  root: ui.slot({ row: true, center: true, gapX: 8 }),
  label: ui.slot({ fontWeight: 700 }),
};

const compact = ui.scope().below('700px', [
  buttonStyles.root({ gapX: 4 }),
  buttonStyles.label({ fontSize: 13 }),
]);
```

That is the important constraint. Custom vocabulary should not stop working
when styles become component-aware.

## Engineering Payoff

Fluentic treats customization as part of the builder layer, not as a separate
runtime-only escape hatch:

- TypeScript sees custom fields and selector arguments.
- The runtime resolves custom-built styles like normal styles.
- Static extraction can evaluate supported custom builders when they are
  configured as import sources.
- Slots, scopes, and component themes can share the same vocabulary as local
  element styles.

The design-system language stays close to React code, but the output remains
CSS.
