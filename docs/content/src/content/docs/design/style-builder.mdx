---
title: Style Builder
description: Start with style declarations as data, then let the builder add behavior without hiding the CSS.
---

Fluentic starts from the smallest thing a frontend developer already writes: a
style object.

```ts
const button = style({
  border: 0,
  borderRadius: 8,
  padding: '8px 12px',
});
```

That object is plain data. You can move it between files, infer it, transform
it, merge it, extract it, and inspect it.

The builder adds the behavior around that data:

```ts
const button = style({
  border: 0,
  padding: '8px 12px',
}).hover({
  opacity: 0.88,
}).media('(max-width: 700px)', {
  width: '100%',
});
```

The declarations stay declarations. Hover and media are not hidden inside the
object; they are chain steps with their own typed API.

## Why This Shape

CSS-in-TypeScript can quickly become one object that describes several
different things at once:

```ts
{
  color: 'white',
  ':hover': {
    color: 'black',
  },
  '@media (max-width: 700px)': {
    padding: 8,
  },
}
```

That shape is familiar and useful. Fluentic takes a slightly different path:
keep declarations as a plain object, and put conditions, selectors,
relationships, and merges in the builder chain.

```txt
declarations        -> style object
self state          -> chain method
responsive rule     -> chain method
component part      -> slot
component override  -> scope
component resolve   -> combineStyle
```

The goal is not novelty. The goal is to keep each styling concern visible
enough that TypeScript, the runtime, and the compiler can all understand it.

## What The Builder Gives

The builder is also where Fluentic becomes customizable.

The default `style` builder understands CSS properties and the default selector
vocabulary:

```ts
style({ color: 'white' })
  .hover({ color: 'black' })
  .focusVisible({ outline: '2px solid currentColor' })
  .select('&[data-state="open"]', { opacity: 1 });
```

But the same builder system can create a project-specific style function with
its own fields, selectors, validation, and priority behavior.

That matters because a design system often grows its own language: `stack`,
`gapX`, `tone`, `pressed`, `open`, `below`, `selected`. Fluentic treats that
language as something the builder can model directly, instead of asking every
component to repeat the same strings and transforms by convention.

## Engineering Payoff

Because styles are builder data, the same value can keep working through
different parts of the system:

- the React runtime can resolve it dynamically;
- the compiler can extract it when it is static;
- DevTools/debug output can keep a path back to the authored value;
- custom builders can still produce ordinary CSS rules;
- slots and scopes can use the same custom vocabulary as local styles.

That is the first design constraint: authoring should feel fluent, but the
result must stay structured enough for runtime composition and build-time
extraction.
