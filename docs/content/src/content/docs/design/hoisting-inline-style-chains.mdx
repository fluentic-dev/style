---
title: Hoisting Inline Style Chains
description: Why inline style chains can still compile to stable extracted output.
---

Module-scope styles are the clearest default:

```ts
const button = style({
  border: 0,
  padding: '8px 12px',
});
```

They are easy to name, easy to reuse, and obviously stable.

But Fluentic also supports a useful production optimization: supported inline
style chains can be hoisted during extraction.

That means this source can still compile into extracted style data:

```tsx
export function Button() {
  const localStatic = style({
    padding: 4,
  }).hover({
    padding: 8,
  });

  return <button css={localStatic}>Save</button>;
}
```

The author wrote the style near the JSX.

The production build can move the generated CSS rule and extracted style object
out of the render path.

## Why This Exists

Inline styles are sometimes the most readable place to put a local style:

- a one-off wrapper inside a component;
- a local state style used only once;
- a small style that is easier to read next to the JSX it affects;
- generated code where creating a top-level name would add noise.

Hoisting lets the compiler keep that authoring style without turning the
production render path into repeated style-object construction.

## What The Compiler Can Hoist

The compiler can hoist supported `style(...)`, `style.slot(...)`, `style.scope(...)`,
and merged chains when it can understand the chain.

The basic snapshot in `examples/snapshots/data/001-basic` shows this shape.

Source:

```tsx
export default ({ color = 'purple' }: { color?: string }) => {
  const localStatic = style({
    padding: 4,
  }).hover({
    padding: 8,
  });

  const localDynamic = style({
    color,
    backgroundColor: 'white',
  }).hover({
    borderColor: color,
  }).merge(dynamicStyle);

  return <div css={[localStatic, localDynamic]} />;
};
```

Compiled output, simplified:

```tsx
const _fluenticStyle = createExtractedStyle([
  ['...', 'padding-class'],
  ['...', 'padding-hover-class'],
]);

const _fluenticToken = createExtractedToken('...', null);

const _fluenticStyle2 = createExtractedStyleMerge(
  [
    ['...', 'color-class', [1, '--var-color', _fluenticToken, 1]],
    ['...', 'background-white-class'],
    ['...', 'border-hover-class', [1, '--var-border', _fluenticToken, 1]],
  ],
  dynamicStyle,
);

export default ({ color = 'purple' }) => {
  const localStatic = _fluenticStyle;
  const localDynamic = withTokens(_fluenticStyle2, [
    _fluenticToken(color),
  ]);

  return <div css={[localStatic, localDynamic]} />;
};
```

The static rule data is created outside render.

The dynamic value is carried as a token binding, so the extracted rule can stay
stable while the render supplies the current value.

## What Hoisting Is Not

Hoisting is not a reason to hide every style inside JSX.

Prefer module-scope styles when the style is:

- reused by more than one render path;
- part of a component's public slot or theme surface;
- a named variant like `dangerButton` or `compactCard`;
- something another developer will search for by name.

Inline chains are best when the style is truly local and reading it beside the
JSX makes the component easier to understand.

## Cache Impact

Without extraction, creating new style objects during render gives the runtime
new identities to walk.

With extraction and hoisting, supported inline chains become stable generated
objects, so repeated component instances can still share resolver/cache paths.

That is why this source shape is allowed:

```tsx
const localStatic = style({
  padding: 4,
}).hover({
  padding: 8,
});
```

But the everyday recommendation remains simple:

```txt
shared style or variant -> module scope
local one-off style     -> inline is okay when extraction can hoist it
open-ended value        -> token/value binding
```

## Disabling Hoist

Hoisting is enabled by default in the integrations that support extraction.

You can turn it off with plugin options:

```ts
stylePlugin({
  hoist: false,
});
```

That is mostly useful for debugging the transform or testing output shape.
Production apps usually want hoisting enabled.
