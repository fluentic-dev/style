---
title: Chains And Selectors
description: Use chain methods to turn states, selectors, and responsive rules into typed styling vocabulary.
---

A style object often starts simple.

Then the element needs hover, focus, media queries, supports rules, or a custom
selector.

Those are not new declarations. They describe when declarations apply.

The chain is not mainly a syntax preference. It is the place Fluentic puts
behavior, so declaration objects can stay plain.

```ts
const button = style({
  color: 'white',
}).hover({
  color: 'black',
}).media('(max-width: 700px)', {
  width: '100%',
});
```

Hover, focus, media, supports, container, and custom selectors all describe when
declarations apply. They are not declarations themselves.

Moving them into chain methods keeps the object plain and gives each behavior a
typed API surface.

## Local Selectors

For a one-off case, a local selector can stay local:

```ts
const item = style({
  opacity: 0,
}).select('&[data-state="open"]', {
  opacity: 1,
});
```

That is enough when the selector belongs to this component.

## Shared Selector Vocabulary

When the same selector becomes part of a design-system language, the builder
can name it:

```ts
const selectors = {
  hover: selector(':hover'),
  open: selector('[data-state="open"]'),
  tone: selector('[data-tone="$"]', assertEnumSelector('neutral', 'danger')),
  below: selector('@media (max-width: $$)', 'media'),
};
```

Now the component code can use that vocabulary directly:

```ts
const item = ui({
  opacity: 0,
}).open({
  opacity: 1,
}).tone('danger', {
  color: '#b91c1c',
}).below('700px', {
  padding: 8,
});
```

The selector strings stay centralized. The method arguments can be typed and
validated. The compiler can understand the same selector model during
extraction.

## Why It Matters

Custom chains let teams choose the level of abstraction they actually use:

```txt
one-off selector       -> .select(...)
shared component state -> .open(...)
shared breakpoint      -> .below(...)
validated tone         -> .tone('danger', ...)
```

The benefit is not that every project needs a custom selector vocabulary. The
benefit is that when a project has one, the vocabulary can be real TypeScript
API instead of repeated selector strings.
