---
title: When Styles Cross Boundaries
description: Reusable components often need a clearer boundary between contract and implementation.
---

You publish a reusable `Card`.

Another team wants the title to look different in their product.

If the public styling surface is a class name, they may reach for a DOM
question:

```css
.product-card > div:first-child h2 {
  font-weight: 760;
}
```

That selector may be the right local choice.

It also knows the wrapper. It knows the heading. It knows the nesting. If the
component changes shape later, the selector may have been depending on more
implementation than anyone intended.

Teams have many reasonable ways to manage this:

- `className` for the root;
- child class props;
- `slotProps`;
- variant props;
- theme override objects.

Each solves a real problem.

The pressure appears when a component spreads across products and nobody is
quite sure which styling details are public and which are accidental.

One possible answer is small:

> The component names the parts it is willing to let others style.

External styling can then target those parts by identity instead of by DOM
path.

In Fluentic, those named parts are slots:

```ts
import { style } from '@fluentic/style';

export const cardStyles = {
  root: style.slot({ borderRadius: 16 }),
  title: style.slot({ fontWeight: 700 }),
  body: style.slot({ lineHeight: 1.6 }),
};
```

Outside code can now talk about `cardStyles.title` instead of
`.product-card > div:first-child h2`.

That does not remove CSS power. It moves the contract from the DOM path to the
component API.

Once those parts have names, styling can cross the component boundary through a
contract instead of a DOM path.
