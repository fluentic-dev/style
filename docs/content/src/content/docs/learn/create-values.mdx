---
title: Create Values
description: Use literal app-wide colors, borders, spacing, and other values as themeable values.
---

`createTokens()` is best when names are part of the theming API.

`createValues()` is for app code that wants to keep using literal values,
especially colors.

The main benefit is practical: you can see the color or spacing value where it
is used, add an optional label when a name helps, and avoid maintaining a
separate token-name map for every app value.

It solves the main inconvenience of named tokens in application code:

```txt
tokens.color.border -> semantic name, literal is elsewhere
color('#d8e3df')    -> literal is visible where it is used
```

This is useful for app-wide colors, border colors, shadows, spacing, and other
values where the literal is easier to read than another token name.

Values can also carry lightweight labels:

```ts
color('#d8e3df | Border')
```

The text before `|` is the default CSS value.

The full string is the identity Fluentic tracks.

That means you do not have to remember whether the token is called
`borderSubtle`, `outlineMuted`, or `cardBorder`.

The literal stays visible, and the label gives just enough identity when the
theme needs to treat two uses differently.

The difference is especially noticeable while editing:

```ts
borderColor: color('#d8e3df | Border')
backgroundColor: color('#ffffff | Surface')
padding: space(16)
```

You can read the value immediately.

Your editor can show color previews.

Search can find every use of `#d8e3df`.

Adding a value is lightweight: add the literal to the allowed list, then use it.

```ts
import { createTheme, createValues, style } from '@fluentic/style';

export const color = createValues([
  '#f5f7f6 | Canvas',
  '#ffffff | Surface',
  '#ffffff | AccentText',
  '#15211d | Text',
  '#60716a | Muted',
  '#d8e3df | Border',
  '#0f766e | Accent',
  '#d8f3ee | AccentSoft',
]);

export const space = createValues(Number, [4, 8, 12, 16, 24]);
```

Use the literal values directly in styles:

```ts
const page = style({
  backgroundColor: color('#f5f7f6 | Canvas'),
  color: color('#15211d | Text'),
});

const panel = style({
  backgroundColor: color('#ffffff | Surface'),
  border: '1px solid',
  borderColor: color('#d8e3df | Border'),
  padding: space(16),
});

const badge = style({
  backgroundColor: color('#d8f3ee | AccentSoft'),
  color: color('#0f766e | Accent'),
  borderRadius: space(8),
  padding: space(8),
});
```

The app code stays easy to read because the value is visible.

Your editor can still show color previews. Search can find the literal value.
Adding a new allowed color is just adding it to the list.

The value is still themeable:

```ts
export const nightTheme = createTheme([
  color('#f5f7f6 | Canvas', '#101514'),
  color('#ffffff | Surface', '#18211f'),
  color('#ffffff | AccentText', '#06201c'),
  color('#15211d | Text', '#eef8f4'),
  color('#60716a | Muted', '#a6bbb3'),
  color('#d8e3df | Border', '#31443e'),
  color('#0f766e | Accent', '#5eead4'),
  color('#d8f3ee | AccentSoft', '#123d38'),
]);
```

The first argument is the literal value used by app code.

The second argument is the value provided by the theme.

Wire the theme at the app boundary or subtree boundary:

```tsx
const css = combineStyle(appStyles);

return <main css={[nightTheme, css.root]}>{children}</main>;
```

Everything inside that subtree keeps using the same literal-facing values.

Labels solve the case where one light-mode value needs different dark-mode
values in different places.

For example, `#ffffff` can mean both a surface and text on an accent:

```ts
export const color = createValues([
  '#ffffff | Surface',
  '#ffffff | AccentText',
]);

const panel = style({
  backgroundColor: color('#ffffff | Surface'),
});

const primaryButton = style({
  color: color('#ffffff | AccentText'),
});

export const darkTheme = createTheme([
  color('#ffffff | Surface', '#18211f'),
  color('#ffffff | AccentText', '#06201c'),
]);
```

Both entries default to `#ffffff`.

Because the labels are different, the dark theme can change them separately.

For numeric scales, pass `Number`.

Numeric values stay typed as numbers, so they fit properties like `gap`,
`padding`, or `borderRadius`:

```ts
export const space = createValues(Number, [4, 8, 12, 16, 24]);

const panel = style({
  gap: space(12),
  padding: space(16),
});

export const denseTheme = createTheme([
  space(12, 8),
  space(16, 12),
]);
```

Use `createValues()` when:

- app-wide colors or border colors should stay visible in code;
- the literal value is easier to read than a name;
- you want IDE color previews and normal value search;
- adding a value should be lightweight;
- values should be easy to theme later;
- the value belongs to the app surface, not a public component API.

Use named tokens when semantic names are the public API.

For a shared component or design system, names like `surface` and `accentText`
may be clearer than app literals:

```ts
const tokens = createTokens({
  color: {
    surface: '#ffffff',
    accentText: '#ffffff',
  },
});

const nightTheme = createTheme([
  tokens.color.surface('#18211f'),
  tokens.color.accentText('#06201c'),
]);
```

That is a public naming contract.

For app-facing values, labels usually keep the code lighter.

Use `createTokens()` when semantic names are clearer:

```ts
tokens.color.accent
tokens.color.surface
tokens.radius.panel
```

Use `createValues()` when literal app values are clearer:

```ts
color('#0f766e | Accent')
color('#d8e3df | Border')
space(16)
```
