---
title: Publishing Component Styling APIs
description: Export only the slots that are part of the component contract.
---

A reusable `Card` eventually becomes someone else's dependency.

When that happens, styling becomes part of the API.

Which parts are consumers allowed to customize?

Everything?

Probably not.

That makes implementation feel public.

Only the root?

Probably not. Product teams often need title, body, media, or action styling.

Make the internal styles first:

```ts
const styles = {
  root: style.slot({ borderRadius: 16 }),
  title: style.slot({ fontWeight: 700 }),
  body: style.slot({ lineHeight: 1.6 }),
  measurementLayer: style({ position: 'absolute' }),
};
```

Only some entries should become promises callers can depend on.

```ts
import { exposeStyle } from '@fluentic/style/utils';

export const CardStyle = exposeStyle({
  root: styles.root,
  title: styles.title,
  body: styles.body,
});
```

`exposeStyle()` turns that list into the styling API callers can import.

Consumers target the exported slots.

The component keeps implementation details private.

This gives you a clean split:

- internal styles can change with the implementation;
- exposed slots are the supported customization surface;
- callers can customize parts without selecting DOM nodes.

Once that contract exists, another team can describe how those public parts
should look.
