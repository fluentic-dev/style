description: A one-page practical map from style() to slots, scopes, and themes.

This is the full Fluentic path in one page.

The core idea is simple:

```txt
write styles in TypeScript
pass them through css
name component parts with slots
group outside overrides with scopes
use named tokens for component theme APIs
use createValues for app-wide literal values
```

## 1. Start with one element

Use `style()` when one element needs CSS:

```tsx
import { style } from '@fluentic/style';

const button = style({
  border: 0,
  borderRadius: 8,
  cursor: 'pointer',
  padding: '8px 12px',
});

export function SaveButton() {
  return <button css={button}>Save changes</button>;
}
```

This is the smallest useful setup:

```txt
style object -> css prop
```

No slots. No scopes. No theme.

## 2. Merge styles with normal JavaScript

The `css` prop accepts arrays.

Use `&&` for optional styles:

```tsx
<button css={[base, props.danger && danger, props.css]}>
  {props.children}
</button>
```

Use `?:` for alternatives:

```tsx
<button css={[base, props.size === 'sm' ? small : large, props.css]}>
  {props.children}
</button>
```

This is intentionally close to React Native style arrays.

If your component accepts a `css` prop, put it last:

```tsx
<button css={[buttonRoot, props.css]} />
```

That means the component provides the default root style, and the caller can
override the root when needed.

## 3. Keep one style self-contained with chains

When the same element needs hover, focus, or media styles, keep the base style
in the object and put conditions in the chain:

```ts
const button = style({
  border: 0,
  borderRadius: 8,
  padding: '8px 12px',
}).hover({
  opacity: 0.86,
}).focusVisible({
  outline: '2px solid currentColor',
  outlineOffset: 2,
}).media('(max-width: 700px)', {
  width: '100%',
});
```

The object says what the element looks like.

The chain says when that changes.

## 4. Use slots for component parts

A reusable component usually has more than one styleable part.

Use `style.slot()` to name the parts you want callers to customize:

```ts title="button.styles.ts"
import { style } from '@fluentic/style';

export const buttonStyles = {
  root: style.slot({
    alignItems: 'center',
    border: 0,
    borderRadius: 8,
    display: 'inline-flex',
    gap: 8,
    padding: '8px 12px',
  }),
  icon: style.slot({
    flex: '0 0 auto',
  }),
  label: style.slot({
    fontWeight: 700,
  }),
};
```

A slot is a public styling target.

The DOM can change later, but `root`, `icon`, and `label` remain the names
callers use.

Callers do not usually pass raw slot calls around.

They put slot overrides inside `style.scope([...])`.

## 5. Use scopes when one idea changes several slots

A variant, state, product skin, or parent interaction often needs to affect
multiple slots together.

Use `style.scope()` to group those slot overrides:

```ts
export const dangerButton = style.scope([
  buttonStyles.root({
    backgroundColor: '#dc2626',
    color: 'white',
  }),
  buttonStyles.icon({
    color: 'white',
  }),
]);

export const compactButton = style.scope([
  buttonStyles.root({ padding: '4px 8px' }),
  buttonStyles.label({ fontSize: 13 }),
]);

export const parentHover = style.scope().hover([
  buttonStyles.root({ transform: 'translateY(-1px)' }),
  buttonStyles.icon({ opacity: 0.8 }),
]);
```

A scope is a package of overrides for public slots.

The scope says what should change. The component decides where those slot
styles attach.

Use a scope even for a single slot override:

```ts
const strongLabel = style.scope([
  buttonStyles.label({ fontWeight: 800 }),
]);
```

## 6. Resolve component styles with combineStyle

Use `combineStyle()` inside the component before rendering JSX:

```tsx
import {
  combineStyle,
  type StyleProp,
  type StyleTheme,
} from '@fluentic/style';

type ButtonProps = {
  danger?: boolean;
  compact?: boolean;
  css?: StyleProp;
  theme?: StyleTheme;
  icon?: React.ReactNode;
  children: React.ReactNode;
};

export function Button(props: ButtonProps) {
  const css = combineStyle(
    buttonStyles,
    props.danger && dangerButton(buttonStyles.root),
    props.compact && compactButton(buttonStyles.root),
    props.theme?.(buttonStyles.root),
  );

  return (
    <button css={[css.root, props.css]}>
      {props.icon ? <span css={css.icon}>{props.icon}</span> : null}
      <span css={css.label}>{props.children}</span>
    </button>
  );
}
```

`combineStyle()` takes the component's base slots plus variants and incoming
themes.

It returns the resolved style for each slot: `css.root`, `css.icon`,
`css.label`.

Then the JSX uses those values:

```tsx
<button css={[css.root, props.css]}>
  <span css={css.icon}>{props.icon}</span>
  <span css={css.label}>{props.children}</span>
</button>
```

## 7. Use bindScope as shorthand

Calling every scope with the same slot is clear:

```tsx
const css = combineStyle(
  buttonStyles,
  props.danger && dangerButton(buttonStyles.root),
  props.compact && compactButton(buttonStyles.root),
  props.theme?.(buttonStyles.root),
);
```

When that list grows, `bindScope()` is the utility shorthand:

```tsx
const css = combineStyle(
  buttonStyles,
  bindScope(
    buttonStyles.root,
    props.danger && dangerButton,
    props.compact && compactButton,
    props.theme,
  ),
);
```

`bindScope()` is only shorthand for calling each scope with
`buttonStyles.root`.

## 8. Use createToken for one shared value

Some changes are not about component parts.

They are shared values: brand color, text color, surface color, focus ring,
spacing, motion, radius.

Use `createToken()` when one value needs to be themeable:

```ts
import { createTheme, createToken, style } from '@fluentic/style';

const accent = createToken('#2563eb');

const link = style({
  color: accent,
});

export const darkTheme = createTheme([
  accent('#60a5fa'),
]);
```

Styles read the token. Themes provide different values for the token.

## 9. Use createTokens for named theme APIs

Use `createTokens()` when the name is part of the theme contract:

```ts
const cardTokens = createTokens({
  color: {
    surface: '#ffffff',
    text: '#111827',
    border: '#d8e3df',
    accent: '#2563eb',
  },
  radius: {
    panel: 8,
  },
});

const cardStyles = {
  root: style.slot({
    backgroundColor: cardTokens.color.surface,
    borderColor: cardTokens.color.border,
    borderRadius: cardTokens.radius.panel,
    color: cardTokens.color.text,
  }),
};

export const darkTheme = createTheme([
  cardTokens.color.surface('#111827'),
  cardTokens.color.text('white'),
  cardTokens.color.border('#374151'),
  cardTokens.color.accent('#60a5fa'),
]);
```

This is good for design systems and shared UI libraries because consumers can
theme named parts like `surface`, `border`, and `accent`.

The object can be nested, so the public theme API can read like
`cardTokens.color.border` or `cardTokens.radius.panel`.

The tradeoff is that literals are hidden behind names. App code may need to
jump to the token file to see what `cardTokens.color.border` actually is.

## 10. Use createValues for app-facing values

Use `createValues()` when literal app values should stay visible and still be
themeable:

```ts
const color = createValues([
  'white',
  '#111827',
  '#d8e3df',
  '#2563eb',
]);

const space = createValues([4, 8, 12, 16, 24]);

const card = style({
  backgroundColor: color('white'),
  borderColor: color('#d8e3df'),
  color: color('#111827'),
  padding: space(16),
  gap: space(8),
});

export const darkRoomyTheme = createTheme([
  color('white', '#111827'),
  color('#111827', 'white'),
  color('#d8e3df', '#374151'),
  space(16, 24),
  space(8, 12),
]);
```

This is useful because app code shows the actual values:

```txt
backgroundColor: color('white')
borderColor: color('#d8e3df')
padding: space(16)
```

The naming burden stays low, color previews still work, and search can find
the literal value.

Use `createTokens()` when names are the API.

Use `createValues()` when literal values make app code easier to write, read,
preview, search, and theme.

Both produce token-backed values. Both can be overridden with `createTheme()`.

## 11. Use createTheme for subtree value overrides

`createTheme()` groups token overrides:

```ts
export const darkRoomyTheme = createTheme([
  color('white', '#111827'),
  color('#111827', 'white'),
  color('#d8e3df', '#374151'),
  space(16, 24),
]);
```

Apply the theme to the element that should define the subtree:

```tsx
const css = combineStyle(appStyles);

return <main css={[darkRoomyTheme, css.root]}>{children}</main>;
```

Token themes change shared values.

Component scopes change component slots.

Real applications usually use both.

## 12. Choose The Smallest Tool

Use this map when deciding what to write:

```txt
one element needs CSS                  -> style()
one element needs hover/media          -> style().hover().media()
caller needs to style root             -> css prop
component exposes styleable parts       -> style.slot()
caller overrides component parts        -> style.scope([...slot(...)])
variant/theme changes one or more parts -> style.scope()
component has slots + themes + states  -> combineStyle()
many scopes attach from the same slot  -> bindScope()
one shared value should be themeable   -> createToken()
named theme API                        -> createTokens()
literal app-wide values should theme   -> createValues()
subtree needs different token values   -> createTheme()
```

The practical rule:

Start with `style()`.

Move to slots, scopes, and tokens only when the styling problem actually needs
them.
