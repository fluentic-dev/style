---
title: Tokens
description: Use named tokens when the name is part of the theming API.
---

Tokens are named theme hooks.

Use `createTokens()` when the name matters to the API. This is common for
component libraries and design systems, where consumers should override names
like `surface`, `text`, `border`, or `accent` instead of raw literals.

Named tokens are good when the name is the contract.

The token shape can be nested.

That lets a component library publish a small theme API, or a design system
publish a larger token tree, without flattening everything into one long list.

The tradeoff is that the literal value is hidden behind a variable. When app
code reads `cardTokens.color.border`, you may need to jump to the token file to
see the actual color.

```ts
import { createTheme, createTokens, style } from '@fluentic/style';

export const cardTokens = createTokens({
  color: {
    surface: '#ffffff',
    surfaceRaised: '#eef4f1',
    text: '#15211d',
    muted: '#60716a',
    border: '#d8e3df',
    accent: '#0f766e',
    accentSoft: '#d8f3ee',
    accentText: '#ffffff',
  },
  radius: {
    panel: 8,
    control: 7,
    pill: 999,
  },
  space: {
    panel: 18,
    row: 10,
  },
  shadow: {
    panel: '0 18px 46px rgba(26, 44, 38, 0.10)',
  },
});
```

Use those names inside component styles:

```ts
export const cardStyles = {
  root: style.slot({
    backgroundColor: cardTokens.color.surfaceRaised,
    border: '1px solid',
    borderColor: cardTokens.color.border,
    borderRadius: cardTokens.radius.panel,
    boxShadow: cardTokens.shadow.panel,
    color: cardTokens.color.text,
    display: 'grid',
    gap: cardTokens.space.panel,
  }),
  badge: style.slot({
    backgroundColor: cardTokens.color.accentSoft,
    borderRadius: cardTokens.radius.pill,
    color: cardTokens.color.accent,
  }),
  title: style.slot({
    color: cardTokens.color.text,
  }),
  text: style.slot({
    color: cardTokens.color.muted,
  }),
};
```

Now a theme can override the component's named values:

```ts
export const darkCardTheme = createTheme([
  cardTokens.color.surfaceRaised('#21302c'),
  cardTokens.color.text('#eef8f4'),
  cardTokens.color.muted('#a6bbb3'),
  cardTokens.color.border('#31443e'),
  cardTokens.color.accent('#5eead4'),
  cardTokens.color.accentSoft('#123d38'),
  cardTokens.shadow.panel('0 22px 52px rgba(0, 0, 0, 0.34)'),
]);
```

Apply the token theme where the component renders:

```tsx
export function CardExample() {
  const css = combineStyle(cardStyles);

  return (
    <article css={[darkCardTheme, css.root]}>
      <h2 css={css.title}>Revenue</h2>
    </article>
  );
}
```

Use named tokens when:

- the value name is part of the public theme contract;
- component consumers should override `surface`, `border`, or `accent`;
- two values need different public semantic names;
- a design system wants semantic names instead of raw literals;
- nested groups like `color`, `radius`, `space`, and `shadow` help consumers
  understand the theme surface.

For app-wide code, that can be less convenient. App teams often want to see
the literal color or spacing value in the style they are editing, with optional
labels only where identity helps.

That is why `createValues()` exists.

Tokens can also be dynamic.

If a prop chooses the actual value, pass a token override to
`combineStyle()`:

```tsx
import { combineStyle, createToken, style } from '@fluentic/style';

const tone = createToken('#0f766e');

const styles = {
  root: style.slot({
    borderColor: tone,
  }),
};

export function Notice(props: { tone?: string }) {
  const css = combineStyle(
    styles,
    props.tone && tone(props.tone),
  );

  return <section css={css.root}>...</section>;
}
```

Use named tokens when the code should read like this:

```txt
cardTokens.color.accent
cardTokens.color.surface
cardTokens.radius.panel
```
