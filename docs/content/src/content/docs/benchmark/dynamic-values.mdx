---

Use the dynamic-values benchmark when the workload is an open-ended value, such
as a user-chosen color, opacity, transform, or shadow width.

```sh
pnpm bench:dynamic-values
```

The main lane compares the recommended stable-rule dynamic value path across
libraries. In Fluentic app code, that means tokens/value bindings. The generated
CSS may use variables internally, but app code should not hand-wire raw custom
properties for normal Fluentic styling.

## Why This Lane Exists

Known variants and open-ended values are different problems.

Known variants should usually become static styles, class switches, recipes, or
scopes. Open-ended values should usually flow through token/value bindings so
the app does not generate a new CSS rule for every value.

The benchmark keeps Fluentic inline dynamic style generation as an intentional
cliff case. It is useful because it shows what happens when an app asks the
runtime to create lots of unique rules.

```sh
ITEMS=4000 BENCH_OUT_DIR=./tmp/dynamic-report pnpm bench:dynamic-values
```

the inline dynamic style case as the recommended Fluentic path.
