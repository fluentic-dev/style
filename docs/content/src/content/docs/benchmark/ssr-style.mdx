description: Server render class and style resolution benchmark report.

Use this benchmark to measure style-resolution work that happens during
`renderToString()` after Fluentic source code has been compiled to production
extracted CSS output.

```sh
pnpm bench:ssr-style
```

Fluentic does not collect server style tags in this benchmark. Its production
SSR path extracts CSS at build time, then resolves class names, css props,
scopes, token overrides, and style prop merges while React renders.

Although this suite runs through SSR, the measured Fluentic work is ordinary
JavaScript style resolution rather than a server-only stylesheet API. The
browser render path uses the same no-hook `combineStyle()` / css-prop resolve /
cache model after CSS has been extracted, so these numbers are also useful for
understanding Fluentic's render-time style overhead in the browser. They do not
measure hydration, style recalculation, layout, paint, or CSS loading.

These are warm steady-state render timings. The benchmark keeps the same Node
process and imported modules across iterations, so Fluentic's runtime caches,
and any comparable library caches, can warm like they do in a long-lived server
or browser app. Row count controls how many instances render per request; it is
not a cache-reset boundary. For the underlying Fluentic resolver and cache
model, see [Style Resolution And Cache](../../design/style-resolution-and-cache/).

The Fluentic and StyleX fixtures are written as normal source and compiled
before the benchmark imports them. Fluentic uses public authoring patterns:
stable `style` objects, slots, and scopes are declared outside render; direct
cases pass direct css props or css arrays with conditional entries; scoped cases
use `combineStyle(styles, bindScope(...))` with slot overrides; and token cases
pass token overrides as additional `combineStyle()` arguments.

## Scenarios

| Scenario | What it measures | Baseline |
| --- | --- | --- |
| Dashboard | One table/dashboard render: page shell, nav, rows, cells, and badges. Each comparable row receives per-row dynamic style variables through `style`. | Plain React with className strings plus the same per-row style object. |
| Composition | Repeated card-like component instances with variants, child slots, class/css composition, and dynamic style variables. | Plain React with manual className composition and style object merging. |

## Case Shapes

Direct css prop:

```tsx
<tr css={[styles.row, active && styles.rowActive]} style={rowVars} />
```

Scoped composition:

```tsx
const css = combineStyle(
  styles,
  bindScope(styles.item, statusScope, compact && compactScope, active && activeScope),
);
```

Token overrides:

```tsx
const css = combineStyle(
  styles,
  bindScope(styles.item, statusScope, compact && compactScope, active && activeScope),
  accentToken(accent),
  surfaceToken(surface),
  ringToken(ring),
);
```

Token override cases use a small set of repeated token values, like real status
or theme variants. High-cardinality per-row values stay on `style`, so the
benchmark does not turn ever-changing values into token overrides.

## Reading The Numbers

`vs baseline` compares each scenario against that group's baseline at the same
row count. Values below `1.00x` are faster than baseline; values above `1.00x`
are slower.

The dashboard baseline is the plain React className plus style variables case.
The className-only row is kept as a reference because it shows the cost of the
dynamic style object itself.

The composition baseline is plain React manual className composition plus style
object merging. This is the closest non-library equivalent to what the
component-style cases do.

Token override rows are intentionally separate. They add CSS variable output
from repeated token values, then merge that output with the normal user
`style` prop when both target the same element. That final style-object merge is
the remaining measurable token cost; the `combineStyle()` wrapper and css-prop
walk are cached for repeated token/scope combinations.

## What It Is Not

This is not a full page delivery benchmark. It does not measure network
delivery, browser hydration, CSS loading, streaming, or third-party server
stylesheet collection APIs.
