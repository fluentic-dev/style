Use the contract benchmark before trusting timing numbers.

```sh
pnpm bench:contract
```

This report builds each benchmark app, verifies that the rendered DOM and
computed styles match the expected contract, records bundle/CSS output size, and
then runs timings only for apps that pass correctness.

## What It Checks

| Check | Why it matters |
| --- | --- |
| Production build | The benchmark should measure production output, not dev-server behavior. |
| DOM structure | Apps need to render the same shape before timing comparisons make sense. |
| Computed styles | The visible result needs to match, not only the component tree. |
| JS and CSS bytes | Build output can regress even when timing stays steady. |
| Style tags and CSS rules | Runtime insertion behavior should stay visible in the report. |

## Filtering

```sh
APP=fluentic-style-extract-chain pnpm bench:contract
APP=@benchmark/app-emotion pnpm bench:contract
SKIP_FLUENTIC_STYLE=1 pnpm bench:contract
INCLUDE_EXPERIMENTAL=1 pnpm bench:contract
```

The `APP` filter accepts benchmark app names and package filters. Use it when a
single app fails and you want a tight repair loop.
