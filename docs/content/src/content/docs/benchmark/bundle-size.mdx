Fluentic includes a bundle report script for the package entry points that
matter to app builds.

Run it from the workspace root:

```sh
pnpm --filter @fluentic/style build
pnpm --filter @fluentic/style report:bundle
```

The report builds the package, follows the compiled dependency graph for each
entry, then creates a small minified bundle for the imported public surface.

## Reading The Columns

| Column | Meaning |
| --- | --- |
| `graph raw` | Total size of compiled files reachable from the entry before bundling. |
| `graph gzip` | Gzip size of those reachable files. |
| `min raw` | Size of the generated minified bundle for the selected public imports. |
| `min gzip` | Gzip size of that generated minified bundle. |

## What To Compare

For an app using static extraction, the most relevant client-side entries are:

| Entry | Why it matters |
| --- | --- |
| extracted production runtime | Runtime helpers used by extracted styles. |
| extracted jsx runtime | JSX runtime used with extracted production output. |
| extracted generated helpers | Helpers used by generated extracted modules. |

For runtime-only app styling, compare `production runtime`.

For React Server Components, compare `server production runtime` or `server
extracted production runtime`, depending on whether your build extracts styles.

Development entries include debug metadata, sourcemaps, element markers, and
runtime inspection helpers. They are useful for development ergonomics, but they
are not the production budget to optimize first.
