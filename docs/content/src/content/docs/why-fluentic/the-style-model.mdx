---
title: The Style Model
description: Learn the small set of ideas behind Fluentic styling.
---

After a few React styling problems, the same pattern keeps showing up.

One element needs declarations.

A reusable component needs stable public parts.

An outside theme needs to customize public parts without knowing DOM.

The component needs one place where props, defaults, themes, and overrides turn
into the final `css` props it renders.

Those situations suggest a small set of front-end building blocks:

```txt
style -> slot -> scope -> resolve
```

Each piece answers one question.

| Idea | Front-end question |
| --- | --- |
| Style | What does this element look like? |
| Slot | Which component parts can callers style on purpose? |
| Scope | Which variant, theme, or state changes those public parts together? |
| Resolve | How do props, defaults, and overrides become final `css` props? |

Most of the other pieces support that path.

Tokens give shared values identity.

Themes override tokens or public component parts.

The compiler can move static styles into CSS during the production build.

The runtime handles values that are still decided by React at render time.

Debugging follows the style names, slots, and scopes you wrote.

The next section turns that shape into practice. It starts with one styled
element, then builds toward reusable components, app themes, debugging, and
component libraries.

Start with [Installation](../learn/installation), then write the smallest
possible style before introducing any component concepts.
