---
title: Style Transforms
description: Add custom style properties and normalize them into CSS.
---

Style transforms let a design system accept its own style shape and compile it
to regular CSS properties.

Use this when you want component authors to write expressive, typed properties
like `row: true`, `column: true`, `gapX`, or token aliases instead of repeating
low-level CSS every time.

## Define A Style Type

Start by extending `CSSProperties` with the custom properties your system owns.

```ts
import type { CSSProperties } from '@fluentic/style';

type LayoutStyle = CSSProperties & {
  row?: boolean;
  column?: boolean;
  gapX?: CSSProperties['columnGap'];
  gapY?: CSSProperties['rowGap'];
};
```

The custom fields are typed everywhere the custom style function is used:
normal styles, slots, scopes, raw styles, plain styles, and keyframes.

## Transform To CSS

`styleTransform(...)` receives your custom style object and returns normal CSS.

```ts
import { styleTransform } from '@fluentic/style';

const layoutTransform = styleTransform<LayoutStyle>({
  transform(style) {
    const { row, column, gapX, gapY, ...css } = style;

    if (row || column) {
      css.display = 'flex';
      css.flexDirection = row ? 'row' : 'column';
    }

    if (gapX !== undefined) css.columnGap = gapX;
    if (gapY !== undefined) css.rowGap = gapY;

    return css;
  },
});
```

This lets authors write:

```ts
const toolbar = layoutStyle({
  row: true,
  gapX: 8,
  alignItems: 'center',
});
```

and emit CSS like:

```txt
display: flex
flex-direction: row
column-gap: 8px
align-items: center
```

## Create The Style Function

Pass the transform to `createStyleFn`.

```ts
import {
  createStyleFn,
  type,
} from '@fluentic/style';
import { MinimalPrioritySelectors } from '@fluentic/style/selector';

export const { style: layoutStyle } = createStyleFn({
  style: type<LayoutStyle>,
  selectors: MinimalPrioritySelectors,
  transform: layoutTransform,
});
```

The resulting `layoutStyle` function has the same features as the default
`style` export, but with your custom properties.

```ts
const card = {
  root: layoutStyle.slot({
    column: true,
    gapY: 12,
    padding: 16,
  }),
  actions: layoutStyle.slot({
    row: true,
    gapX: 8,
    justifyContent: 'end',
  }),
};
```

## Transform Slots And Scopes

The transform runs for slot styles and scope overrides too.

```ts
const compact = layoutStyle.scope().media('(max-width: 700px)', [
  card.root({
    gapY: 8,
    padding: 12,
  }),
  card.actions({
    column: true,
    gapY: 6,
  }),
]);
```

That keeps design-system shortcuts available in responsive and scoped
overrides, not only in base styles.

## Transform Raw And Plain Styles

Custom style functions also expose `raw(...)` and `plain(...)`.

Use `raw(...)` for reusable style objects that should accept the full Fluentic
style value model. That includes design tokens, at-rule value refs, and value
weight tuples.

```ts
import { createToken } from '@fluentic/style';

const surface = createToken('Canvas');

const reset = layoutStyle.raw({
  row: true,
  gapX: 4,
  backgroundColor: surface,
  color: [1, 'CanvasText'],
});
```

Use `plain(...)` for reusable objects that should stay close to the plain
TypeScript style shape. It is useful for constants you want to spread into
styles without accepting Fluentic token/value-ref extensions.

```ts
const object = layoutStyle.plain({
  column: true,
  gapY: 10,
});
```

Both helpers return the object you pass in. They are type helpers for reusable
style fragments:

```ts
const card = layoutStyle({
  ...reset,
  ...object,
  padding: 16,
});
```

In short:

```txt
raw(...)    reusable Fluentic style object; supports tokens and value refs
plain(...)  reusable plain style object; no token or value-ref extensions
```

Value weight tuples are accepted by the current plain style type. Use project
conventions if you want `plain(...)` fragments to remain completely unweighted.

## Pattern

Good transforms are small and predictable:

```txt
custom typed property
  -> remove it from the object
  -> write one or more CSS properties
  -> return the remaining CSS object
```

Keep selector behavior separate from transform behavior. Transforms decide
which CSS declarations exist; selectors decide when those declarations apply.
